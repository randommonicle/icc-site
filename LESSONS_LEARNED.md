# ICC Platform — Lessons Learned

Hard-won gotchas and how we resolved them. Read before working near the chat proxy, bookings, email, or deploy config. Each has an ID (L-NNN) so other docs and code comments can reference it.

Some entries are seeded from the existing codebase and from the sibling ASH app, where the same class of problem already bit us. They are recorded here pre-emptively so we do not relearn them.

---

## L-001 — The chat origin check fails OPEN by default
When `ALLOWED_ORIGINS` is not set, `chat.js` derives a default allowlist and lets unrecognised origins through (logging a warning) rather than returning 403. The comment is explicit: the per-IP rate limit is treated as the real defence so a 403 storm never blocks customers. The consequence is that, until `ALLOWED_ORIGINS` is set in Netlify, the rate limit is the only thing standing between an attacker and your Anthropic bill. **Set `ALLOWED_ORIGINS` to the real production origins before launch.** Verify by POSTing from a disallowed origin and confirming a 403 once strict mode is on.

## L-002 — Prompt caching only works if the cache prefix stays stable
`chat.js` sends the system prompt as two blocks: a large static block with `cache_control: ephemeral`, and a small dynamic block (assistant name, today's date, bookable dates). This is deliberate. If per-conversation or per-day content leaks into the cached block, the cache prefix changes every request and the discount evaporates. When editing the system prompt, keep anything variable in the second, uncached block. Cache hits cost roughly 10% of normal input — a large saving over a multi-message conversation.

## L-003 — Render all AI/booking output as inert text, never innerHTML
Both `index.html` and `admin.html` build AI replies and booking fields with DOM nodes (`createTextNode`) or an `escHtml`/`esc` helper, never `innerHTML`. This is the prompt-injection defence: a malicious or model-confused reply must render as text, not execute. The one intentional `innerHTML` use is the static typing-indicator markup. Do not introduce `innerHTML` for any content that originated from the model or a customer.

## L-004 — "Accepted by Resend" is not "delivered"
A 200 from the Resend API means Resend accepted the message, not that it reached the inbox. Actual delivery (delivered/bounced/blocked) lives in the Resend dashboard. Two specifics carried from the ASH app: corporate mail filters can hard-block certain content, and the sandbox sender (`onboarding@resend.dev`) has limited deliverability. Before relying on booking emails in the field, verify a real sending domain in Resend and set `OPERATOR_FROM` / `CUSTOMER_FROM` / `OPERATOR_EMAIL` to real addresses. Confirm an actual delivery, do not trust the API 200.

## L-005 — Netlify functions run in UTC; parse dates as local components
Booking dates are `YYYY-MM-DD` strings. `generateJobCardPDF` parses them into local date components (`new Date(y, m-1, d)`) precisely so the function's UTC runtime can never shift the printed day. Anywhere a date string is turned into a `Date` for display or day-of-week, parse the components explicitly rather than `new Date("YYYY-MM-DD")` (which is interpreted as UTC midnight and can render as the previous day in some timezones).

## L-006 — Rate-limit every expensive or abusable path, not just chat
Originally only `checkChatRateLimit` guarded the AI chat path; `check_availability` and `confirm_booking` had no cap, so a direct POST loop could block out calendar slots (slot-griefing) — `validateBooking` stops malformed payloads but not volume. **Resolved** on `hardening/phase0-pre-launch`: the limiter is now a reusable pure function `rateLimit(store, key, limit, windowMs, now)` (unit-tested in `test/hardening.test.js`) wrapped by `enforceRateLimit`, applied per-IP to all three POST actions — chat 30/hr, `confirm_booking` 5/hr, `check_availability` 60/hr — each failing open if Blobs are unavailable. Keep the rule going forward: any new endpoint that costs money or writes shared state gets a per-IP cap from day one. Caveat: this is per-IP and best-effort (shared/NAT addresses, IP rotation) — it is volume protection, not authentication. The origin allowlist (L-001) and `validateBooking` remain the other layers, and the real end-to-end check (a 429 against live Blobs) only happens on a deployed environment, not locally.

## L-007 — Netlify Blobs needs explicit auth for some deploy types
`getStore` is called with `siteID` and `token` (`NETLIFY_SITE_ID` / `NETLIFY_TOKEN`) because drag-and-drop deploys do not auto-configure the Blobs context. If Blobs reads/writes start failing after a deploy, confirm those two env vars are present in Netlify. The rate-limit and booking code fail open / log on Blobs errors, so a misconfiguration can be silent — check function logs, not just the UI.

## L-008 — Don't let a JSON.parse failure silently drop content
Carried from the ASH app. The booking flow depends on parsing a `BOOKING_READY:{...}` block out of the model's reply, and stored bookings are JSON in Blobs. A truncated or malformed JSON that is caught and swallowed makes a booking silently vanish with the customer none the wiser. Keep `max_tokens` generous for any response carrying JSON, log parse failures, and surface a fallback message ("call us on 01242 279590") rather than failing silently. The client already shows a phone-number fallback on booking error — preserve that pattern as the flow grows.

## L-009 — Verify third-party and marketing claims before repeating them to customers
The Phase 0 assistant prompt repeated a supplier's marketing as fact: "WoolSafe approved", "76 dB / quieter than a vacuum", "EMV 409", "exothermic reaction draws dirt from the base of the fibre", "80% less water". A verified research pass (`docs/TEXATHERM_KNOWLEDGE_BRIEF.md`, on the Phase 1 branch) found the **"WoolSafe approved"** claim indefensible — Texatherm is absent from the WoolSafe directory, carries no mark or certificate number on its own product pages, and WoolSafe does not certify systems/machines/methods at all (only chemical products and accredited *service-provider companies*). The **76 dB** figure was refuted (the TC 170 page lists 63 dB(A)), the **machine model was wrong** (it is the EMV 401), and the performance figures are single-source manufacturer marketing. A Google AI Overview confidently asserted the opposite — it had blended Texatherm's real product names with Texatherm's own "safe for wool" marketing. **Rule:** any third-party, certification, or performance claim shown to a customer must be substantiated from a primary/independent source or attributed explicitly ("the manufacturer states…") — never asserted as fact, and never trusted from an AI summary. The distinction that bit us: a product being **"safe for use on wool"** (a supplier's in-house assessment, fine to state) is NOT the same as **"WoolSafe® approved"** (an independent certification ICC does not hold). The defensible USP is the science we can stand behind (low-moisture genuinely suits wool and natural fibres; residue attracts soil; honest stain limits), not borrowed marketing. `docs/TEXATHERM_KNOWLEDGE_BRIEF.md` is the single source (D-006) for both the site and the assistant — keep them aligned to it.

**Addendum (3 June 2026):** the same drift then surfaced in the *site*. The staged Astro `services.astro` and `history.astro` still asserted "WoolSafe approved", the exothermic "draws dirt from the base" mechanism, and the manufacturer figures as fact, even though PR #2 had already corrected the assistant. Fixing the claim in `chat.js` did **not** fix the duplicate copy elsewhere. **Rule reinforced:** when you correct a third-party or marketing claim, grep the whole repo against the brief (site content, not just the assistant prompt) and fix every occurrence in the same pass. Until D-006 is implemented as one shared source, the same fact lives in several files and drifts independently.

**Addendum (4 June 2026, evening):** the same class of unsubstantiated claim was also sitting in the site's *trust stats* (`500+ Cleans`, `98% Satisfaction`, `98% 5-Star Reviews` on the home and About pages) for a business with no track record yet. Removed on `feat/pre-cutover-polish` and replaced with capability/fact tiles (carpet types, trading days, deposit). The audit caught it by reading the rendered pages, not the assistant prompt, which is the L-009 reflex: **grep the built site, not just `chat.js`.** A grep-able `TODO(trust-stats)` anchor marks where real, substantiated metrics may go once they exist.

## L-010 — The Phase 0 palette was ASH-style blue, not the ICC brand; the dark tokens now hold green
The Astro site (and the live `index.html`) inherited a blue navy base (`#1a3a5c` / `#0d2236`) that reads as ASH branding. The ICC logo is deep green + teal + gold. On 3 June 2026 the Astro chrome was rebranded onto the logo's green (nav, footer, page headers, headings). **Gotcha for the next person:** the canonical brand tokens in `site/src/styles/global.css` are now `--green` / `--green-dark` / `--green-mid`; the old `--navy` / `--navy-dark` / `--navy-mid` names are kept as **aliases** pointing at the greens, so existing `var(--navy*)` references still resolve. If you see `var(--navy)` rendering green, that is why — migrate refs to `--green*` when convenient. Follow-ups: Ben accepted the green provisionally and may want it toned down (change `--green-dark:#0c2c25` and `--green:#15564a` in that one place); the **live `index.html` is still blue** and should get the same treatment before or at cutover (a live change, needs deploy sign-off). Lesson: take brand colours from the actual brand asset (the logo), not a sibling-project template — the ASH/PropOS conventions transfer for engineering discipline, not for visual identity.

**Addendum (4 June 2026):** WCAG AA pass on `feat/pre-cutover-polish`. The teal `--teal:#1a8a7a` was **4.23:1** on white (just under AA 4.5 for small text), so it was darkened to **`#178072` (4.8:1)**; because it is used for both brand accent and text, the one token change fixes section labels, prices, links and the white-on-teal buttons together. Button hovers used to *lighten* to `--teal-light` (white text 2.47:1, a fail) and now **darken** to `--green-mid` (6.4:1). ## L-011 — pgTAP `throws_ok` checks the error MESSAGE, not just the SQLSTATE
The Slice 2 schema test (`supabase/tests/schema_test.sql`) asserts that an overlapping booking is rejected by the `btree_gist` exclusion constraint. Written as `throws_ok(sql, '23P01', 'overlapping non-cancelled job is rejected')` it **failed**: given a third string argument, `throws_ok` treats it as the expected error *message* and compares it to Postgres's actual text. An exclusion-constraint violation emits Postgres's own wording (`conflicting key value violates exclusion constraint "..."`), which never matches the human string, so the assertion failed even though the correct error (SQLSTATE `23P01`) fired. **Fix:** pass `null` as the message (4th arg) so it asserts the SQLSTATE only, and put the human string in the description (5th arg): `throws_ok(sql, '23P01', null, 'overlapping ... is rejected')`. Verified: `supabase test db` passes pgTAP 10/10. Rule: when asserting a specific SQLSTATE in pgTAP, pass `null` for the message unless you genuinely mean to assert Postgres's exact wording.

## L-012 — Netlify builds deploy previews on a fresh commit, not on a PR base-retarget or reopen
Merging a stack of PRs where each targets the one below it (`#8 → #7 → #6 → main`) leaves the upper PRs with **no deploy preview**: Netlify only builds previews for PRs whose base is the production branch (`main`). The instinct is to retarget each PR to `main` as the one below merges, but retargeting via the API emits a GitHub `edited` event, which Netlify does **not** build. Closing and reopening the PR (a `reopened` event) also failed to trigger a build, because the head commit was unchanged and Netlify had never built that SHA as a preview against `main` — there was no `synchronize` event. The reliable triggers are a **fresh commit** on the branch (`synchronize`), or a manual `netlify deploy` from a checkout. On 7 June 2026, rather than dirty the history with throwaway commits, we verified Slices 1 and 2 **directly on production after merge**: Netlify atomic deploys never swap in a failed build (production keeps serving the last good deploy), `node --test` had already proved the modules load, and an immediate `/api/chat` + `/api/bookings` probe confirmed the live functions, with `git revert` as the undo. **Rule:** for a stacked-PR merge, decide the verification path up front — either verify on production (atomic-deploy + tests + live probe), or force each preview with a throwaway/empty commit. Do not assume a retarget or reopen produces a preview; it does not.

## L-013 — For the hosted ICC Supabase, use the CLI (account-wide token), not the MCP connector (single-org OAuth)
The Supabase MCP connector authenticates via OAuth scoped to **one organisation**. Ours was bound to the build-lead's personal `randommonicle's Org`, so once the dedicated **Intelligent Carpet Cleaning** org was created (10 June 2026), the MCP's `list_organizations` could not see it and could not create the project there — and re-authorising only lets you pick a single org, which would drop access to the other projects (PropOS, ASH). What works: the **Supabase CLI** authenticates with a **personal access token that is account-wide**, so it sees every org (and was already logged in on the machine). Stand up and drive the hosted ICC project through the CLI from the repo (`supabase link --project-ref qzcfgpfvzpynnjgriqqn`, then `db push` / `gen types` / `migration repair`), not the MCP. **Aside (same machine):** Claude runs here via the **desktop app**, so there is no `claude` (or `netlify`) CLI on PATH in a plain terminal — manage MCP connectors in the app's settings, and don't assume terminal CLIs exist.

## L-014 — The browser reads `data.content[0].text` and ignores `stop_reason`, so the server must return a single text block
`index.html` parses the `/api/chat` response as `data.content[0].text` (then scans that text for the `BOOKING_READY:{…}` block) and never inspects `stop_reason`. That contract constrains every server-side enrichment of the chat path. The moment the server runs a tool loop (Slice 4b `escalate_to_human`) or enables Citations (Slice 4c), the model's response is no longer a single text block: a tool round ends on `stop_reason:"tool_use"` with a `tool_use` block in `content`, and Citations splits the reply across multiple text blocks each carrying a `citations` array. Returning that raw shape would make the browser read only the first block (truncating the reply) or `undefined` (a `tool_use` block has no `.text`). **Fix (Slice 4b):** resolve tool rounds server-side (`runAssistantTurn`) and collapse the final message to one text block before returning (`withSingleTextBlock` in `chat.js` — concatenates text blocks, drops `tool_use`, passes API-error objects through untouched). When Slice 4c renders citation links to the customer, the *client* renderer must be extended to consume a structured citations payload (safe DOM `<a>`, sanitised `https`, never `innerHTML` — L-003); until then the normaliser keeps the single-text contract intact. **Rule:** any change that makes Anthropic return multiple or non-text content blocks must be normalised server-side (or the client updated in lockstep) — never assume `content[0].text` is the whole reply.

## L-015 — A wrong/absent operator address silently suppressed every operator email via Resend's account-level list
`OPERATOR_EMAIL` was set to `mark@intelligentclean.co.uk`, but that mailbox **never existed** — Mark's Workspace account was created as `mark_director@intelligentclean.co.uk` (only `ben@` and `mark_director@` appear in the Directory; the "mark@ owner mailbox" in the 10 June notes was aspirational). On 10 June the live TEST booking's **operator** email to `mark@` hard-bounced ("recipient not found"), and Resend added `mark@` to its **account-level suppression list**. From then on every send to `mark@` logged `Sent → Suppressed` in the same second — a 200 from the Resend API, never delivered (the purest form of L-004). Because the booking-confirmation operator leg and the new Slice 4b escalation both target `OPERATOR_EMAIL`, **both were silently dropped**; the 10 June "verified end-to-end" had only ever checked the *customer* email. Caught while preview-verifying #17: the escalation rendered perfectly in Resend but showed **Suppressed**. **Fix:** point `OPERATOR_EMAIL` at the real mailbox (`mark_director@`), remove `mark@` from Resend's suppression list, re-send on a fresh preview → **Delivered**. **Rules:** (1) verify **every** recipient leg of a multi-recipient email flow, not just the customer's; (2) Resend's account-level suppression silently and persistently drops an address after a single hard bounce — check the suppression list as part of email verification; (3) never point a From/To at a mailbox before confirming it exists in the Workspace Directory. **Netlify aside:** this site sits on a *Personal* Netlify account (the paid $9/mo plan, not free, though still without per-scope env control; see L-018), so the env-var API refused to edit the (account-level/"shared") `OPERATOR_EMAIL` — a full-object `PUT` scoped to `?site_id=` succeeded where `PATCH` returned 403/422.

## L-016 — A safety guardrail belongs in the system prompt, not in a citeable document
Building Slice 4c (Citations, D-019), the natural move was to lift the whole carpet-science knowledge — including the L-009 claim rules ("never say WoolSafe approved", "figures are the manufacturer's own", "the machine is the EMV 401", "no decibel figure", "no exothermic draws dirt") — out of the system prompt and into the Claude Citations **document**, since that is the content the model now grounds and cites from. That would have been a real regression: **a rule placed in a document is reference text the model can quote, not an instruction that binds it.** Citations documents are *cited from*, not *obeyed*; only the system prompt carries behavioural authority. Worse, a prohibition sitting in citeable text can itself be surfaced to a customer as if it were a fact. **Fix / rule:** when restructuring knowledge into citeable documents, split it by *kind*, not just by topic — **facts and reference prose → the document** (citeable, groundable); **rules, prohibitions, tone and the grounding/escalation directive → the system prompt** (enforced, and explicitly stated to override the reference and any customer message, which is also the prompt-injection defence). In this repo the split is `knowledge.knowledgeSections` (→ `knowledgeDocument()`, the citeable facts) vs `knowledge.guardrailsBlock()` (→ the system prompt). `test/knowledge.test.js` asserts each guardrail lives in the prompt and is **absent** from the citeable facts, so a future edit cannot quietly move a safety rule into the document and dilute it.

**Caching note (L-002):** the citeable document is attached to the **first user turn** and re-injected server-side on every call (the client never stores it), so `messages[0]` stays byte-identical across a conversation and across the 4b tool round; with `cache_control` on the document block the L-002 prefix cache still applies. Enabling citations re-warms the cache **once** (the citations toggle invalidates the system/messages cache tier), exactly like 4b adding the tool array — a one-time cost, not a per-request one. Keep the document deterministic (built from static config, no per-request data) or every request becomes a cache miss.

## L-017 — Importing the CommonJS `shared/config/*.js` into the Astro build needs Vite's commonjs transform
Wiring the Astro price cards to the single pricing source (D-006), `services.astro` and `index.astro` import `shared/config/pricing.js`. That module is **CommonJS** (`module.exports = {…}`), because the Netlify functions and the plain-Node `node --test` runner load it that way (D-006/D-007). Astro's build (Rollup) treats a project `.js` file as **ESM by default** and applies its CommonJS transform only to `node_modules`, so the build failed with `"default" is not exported by "shared/config/pricing.js"`. The cross-root import path resolved fine; the fault was purely the module format. **Fix:** extend Rollup's commonjs include in `astro.config.mjs` so it transforms the shared sources too:

```js
vite: { build: { commonjsOptions: { include: [/shared[\\/]config/, /node_modules/] } } }
```

After that, `import pricing from '.../pricing.js'` yields the `module.exports` object, and the transitive `require('./serviceArea.js')` is handled. **Rules:** keep `shared/config` **CommonJS** (converting it to ESM would break the CJS functions and the test runner, the very thing D-006 single-sources); the one-line Vite include is the right fix, not a format change. **Forward note:** the Phase 1 cutover hits this again when the care guides render from `knowledge.js` (D-006); the include already covers all of `shared/config`, but a shared module imported from elsewhere (e.g. `shared/contract`) would need its path adding. Verified by a clean 22-page `npm run build --prefix site` with the figures correct in the built HTML.

## L-018 — Netlify env vars via the API on the paid $9 Personal plan: no per-scope control; POST creates, PUT updates, and a redeploy bakes them for functions

Enabling Slice 5a, setting `SUPABASE_URL` / `SUPABASE_SERVICE_ROLE_KEY` through the Netlify env-var API first failed with `403 {"message":"Upgrade your Netlify account to set specific scopes"}` because the body included `scopes:["functions"]`. Per-scope control is a higher-tier feature; the **paid $9/month Personal plan** (this site's plan, correcting L-015's "free-feature" aside) can only set a variable for **all** scopes. **Fix:** omit `scopes` so the var applies to all scopes. That is how the existing secrets are already stored, and it is safe here because nothing in the site *build* references the service-role key, so it never reaches client output (the value lives only in the function runtime, not the deployed assets).

Mechanics confirmed this session, all site-scoped with `?site_id=`:
- **Create:** `POST /api/v1/accounts/{account_slug}/env?site_id={site_id}` with a JSON array of `{key, values:[{value, context:"all"}]}` and **no** `scopes` → `201`.
- **Update one key:** `PUT /api/v1/accounts/{account_slug}/env/{key}?site_id={site_id}` with the single object → `200` (the L-015 method).
- **Redeploy:** `POST /api/v1/sites/{site_id}/builds` triggers a production build.
- **Env changes are baked at deploy:** a function only sees a new or changed value after a redeploy. Verified by flipping `OPERATOR_EMAIL` to a test address, redeploying, confirming, then restoring it with a second redeploy. So "set the var" and "the live function uses it" are two steps.

The `account_slug` comes from `GET /api/v1/sites/{site_id}` (`.account_slug`). Cross-ref L-015 (same Personal-account env API).

## L-019 — Apply a hosted Supabase migration with the Management API when you have the access token but not the DB password

Enabling Slice 5b needed one DDL migration (`alter table jobs alter column postcode drop not null`) on the hosted project, but `supabase db push` and `supabase migration repair` both connect to Postgres directly and so need the **database password**, which was not in the repo `.env` (only `SUPABASE_URL` + `SUPABASE_SERVICE_ROLE_KEY`), and the worktree was not `supabase link`ed. The service-role key is a PostgREST/GoTrue JWT, **not** a Postgres role password, so it cannot run DDL over the REST API either.

**What works:** the **Supabase Management API** query endpoint — `POST https://api.supabase.com/v1/projects/{ref}/database/query` with body `{"query":"<sql>"}` and `Authorization: Bearer $SUPABASE_ACCESS_TOKEN` — runs arbitrary SQL server-side (it is what the dashboard SQL Editor uses). The **account-wide personal access token** is what the CLI authenticates with; here it was already present as `SUPABASE_ACCESS_TOKEN` in the environment, so a one-off `fetch` applied the ALTER (HTTP 201) and a follow-up `select … from information_schema.columns` / `pg_constraint` catalog-verified it (`is_nullable=YES`, invariants intact) — the db-migration-verification check, done remotely.

**Two gotchas:** (1) the Management API **bypasses migration history** (`supabase_migrations.schema_migrations`), so `db push` will later see the migration as pending and re-apply it — fine here because `ALTER … DROP NOT NULL` is **idempotent** (a no-op on an already-nullable column, no error), but for a non-idempotent migration you would corrupt state, so prefer `db push` (clean history) when the DB password is available and only reach for this when it is not, favouring idempotent SQL. Record the migration with `migration repair --status applied <version>` once the password is to hand. (Done for 5b on 14 June: after `SUPABASE_DB_PASSWORD` was added to `.env`, `supabase link` + `migration repair --status applied 20260614100000` reconciled hosted history with no schema change — `migration list` then showed all four migrations Local + Remote.) (2) **Quoting:** running the SQL through `node -e '…'` inside a bash single-quoted command silently strips the SQL's own single quotes (`'public'` → `public` → `column "public" does not exist`); use a quoted heredoc (`node <<'NODE' … NODE`) or a temp script file so string literals survive. The DDL itself (no quotes) ran fine; only the verification `select`s broke until re-run via heredoc.
