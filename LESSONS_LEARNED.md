# ICC Platform — Lessons Learned

Hard-won gotchas and how we resolved them. Read before working near the chat proxy, bookings, email, or deploy config. Each has an ID (L-NNN) so other docs and code comments can reference it.

Some entries are seeded from the existing codebase and from the sibling ASH app, where the same class of problem already bit us. They are recorded here pre-emptively so we do not relearn them.

---

## L-001 — The chat origin check fails OPEN by default
When `ALLOWED_ORIGINS` is not set, `chat.js` derives a default allowlist and lets unrecognised origins through (logging a warning) rather than returning 403. The comment is explicit: the per-IP rate limit is treated as the real defence so a 403 storm never blocks customers. The consequence is that, until `ALLOWED_ORIGINS` is set in Netlify, the rate limit is the only thing standing between an attacker and your Anthropic bill. **Set `ALLOWED_ORIGINS` to the real production origins before launch.** Verify by POSTing from a disallowed origin and confirming a 403 once strict mode is on.

## L-002 — Prompt caching only works if the cache prefix stays stable
`chat.js` sends the system prompt as two blocks: a large static block with `cache_control: ephemeral`, and a small dynamic block (assistant name, today's date, bookable dates). This is deliberate. If per-conversation or per-day content leaks into the cached block, the cache prefix changes every request and the discount evaporates. When editing the system prompt, keep anything variable in the second, uncached block. Cache hits cost roughly 10% of normal input — a large saving over a multi-message conversation.

## L-003 — Render all AI/booking output as inert text, never innerHTML
Both `index.html` and `admin.html` build AI replies and booking fields with DOM nodes (`createTextNode`) or an `escHtml`/`esc` helper, never `innerHTML`. This is the prompt-injection defence: a malicious or model-confused reply must render as text, not execute. The one intentional `innerHTML` use is the static typing-indicator markup. Do not introduce `innerHTML` for any content that originated from the model or a customer.

## L-004 — "Accepted by Resend" is not "delivered"
A 200 from the Resend API means Resend accepted the message, not that it reached the inbox. Actual delivery (delivered/bounced/blocked) lives in the Resend dashboard. Two specifics carried from the ASH app: corporate mail filters can hard-block certain content, and the sandbox sender (`onboarding@resend.dev`) has limited deliverability. Before relying on booking emails in the field, verify a real sending domain in Resend and set `OPERATOR_FROM` / `CUSTOMER_FROM` / `OPERATOR_EMAIL` to real addresses. Confirm an actual delivery, do not trust the API 200.

## L-005 — Netlify functions run in UTC; parse dates as local components
Booking dates are `YYYY-MM-DD` strings. `generateJobCardPDF` parses them into local date components (`new Date(y, m-1, d)`) precisely so the function's UTC runtime can never shift the printed day. Anywhere a date string is turned into a `Date` for display or day-of-week, parse the components explicitly rather than `new Date("YYYY-MM-DD")` (which is interpreted as UTC midnight and can render as the previous day in some timezones).

## L-006 — Rate-limit every expensive or abusable path, not just chat
Originally only `checkChatRateLimit` guarded the AI chat path; `check_availability` and `confirm_booking` had no cap, so a direct POST loop could block out calendar slots (slot-griefing) — `validateBooking` stops malformed payloads but not volume. **Resolved** on `hardening/phase0-pre-launch`: the limiter is now a reusable pure function `rateLimit(store, key, limit, windowMs, now)` (unit-tested in `test/hardening.test.js`) wrapped by `enforceRateLimit`, applied per-IP to all three POST actions — chat 30/hr, `confirm_booking` 5/hr, `check_availability` 60/hr — each failing open if Blobs are unavailable. Keep the rule going forward: any new endpoint that costs money or writes shared state gets a per-IP cap from day one. Caveat: this is per-IP and best-effort (shared/NAT addresses, IP rotation) — it is volume protection, not authentication. The origin allowlist (L-001) and `validateBooking` remain the other layers, and the real end-to-end check (a 429 against live Blobs) only happens on a deployed environment, not locally.

## L-007 — Netlify Blobs needs explicit auth for some deploy types
`getStore` is called with `siteID` and `token` (`NETLIFY_SITE_ID` / `NETLIFY_TOKEN`) because drag-and-drop deploys do not auto-configure the Blobs context. If Blobs reads/writes start failing after a deploy, confirm those two env vars are present in Netlify. The rate-limit and booking code fail open / log on Blobs errors, so a misconfiguration can be silent — check function logs, not just the UI.

## L-008 — Don't let a JSON.parse failure silently drop content
Carried from the ASH app. The booking flow depends on parsing a `BOOKING_READY:{...}` block out of the model's reply, and stored bookings are JSON in Blobs. A truncated or malformed JSON that is caught and swallowed makes a booking silently vanish with the customer none the wiser. Keep `max_tokens` generous for any response carrying JSON, log parse failures, and surface a fallback message ("call us on 01242 279590") rather than failing silently. The client already shows a phone-number fallback on booking error — preserve that pattern as the flow grows.

## L-009 — Verify third-party and marketing claims before repeating them to customers
The Phase 0 assistant prompt repeated a supplier's marketing as fact: "WoolSafe approved", "76 dB / quieter than a vacuum", "EMV 409", "exothermic reaction draws dirt from the base of the fibre", "80% less water". A verified research pass (`docs/TEXATHERM_KNOWLEDGE_BRIEF.md`, on the Phase 1 branch) found the **"WoolSafe approved"** claim indefensible — Texatherm is absent from the WoolSafe directory, carries no mark or certificate number on its own product pages, and WoolSafe does not certify systems/machines/methods at all (only chemical products and accredited *service-provider companies*). The **76 dB** figure was refuted (the TC 170 page lists 63 dB(A)), the **machine model was wrong** (it is the EMV 401), and the performance figures are single-source manufacturer marketing. A Google AI Overview confidently asserted the opposite — it had blended Texatherm's real product names with Texatherm's own "safe for wool" marketing. **Rule:** any third-party, certification, or performance claim shown to a customer must be substantiated from a primary/independent source or attributed explicitly ("the manufacturer states…") — never asserted as fact, and never trusted from an AI summary. The distinction that bit us: a product being **"safe for use on wool"** (a supplier's in-house assessment, fine to state) is NOT the same as **"WoolSafe® approved"** (an independent certification ICC does not hold). The defensible USP is the science we can stand behind (low-moisture genuinely suits wool and natural fibres; residue attracts soil; honest stain limits), not borrowed marketing. `docs/TEXATHERM_KNOWLEDGE_BRIEF.md` is the single source (D-006) for both the site and the assistant — keep them aligned to it.

**Addendum (3 June 2026):** the same drift then surfaced in the *site*. The staged Astro `services.astro` and `history.astro` still asserted "WoolSafe approved", the exothermic "draws dirt from the base" mechanism, and the manufacturer figures as fact, even though PR #2 had already corrected the assistant. Fixing the claim in `chat.js` did **not** fix the duplicate copy elsewhere. **Rule reinforced:** when you correct a third-party or marketing claim, grep the whole repo against the brief (site content, not just the assistant prompt) and fix every occurrence in the same pass. Until D-006 is implemented as one shared source, the same fact lives in several files and drifts independently.

**Addendum (4 June 2026, evening):** the same class of unsubstantiated claim was also sitting in the site's *trust stats* (`500+ Cleans`, `98% Satisfaction`, `98% 5-Star Reviews` on the home and About pages) for a business with no track record yet. Removed on `feat/pre-cutover-polish` and replaced with capability/fact tiles (carpet types, trading days, deposit). The audit caught it by reading the rendered pages, not the assistant prompt, which is the L-009 reflex: **grep the built site, not just `chat.js`.** A grep-able `TODO(trust-stats)` anchor marks where real, substantiated metrics may go once they exist.

## L-010 — The Phase 0 palette was ASH-style blue, not the ICC brand; the dark tokens now hold green
The Astro site (and the live `index.html`) inherited a blue navy base (`#1a3a5c` / `#0d2236`) that reads as ASH branding. The ICC logo is deep green + teal + gold. On 3 June 2026 the Astro chrome was rebranded onto the logo's green (nav, footer, page headers, headings). **Gotcha for the next person:** the canonical brand tokens in `site/src/styles/global.css` are now `--green` / `--green-dark` / `--green-mid`; the old `--navy` / `--navy-dark` / `--navy-mid` names are kept as **aliases** pointing at the greens, so existing `var(--navy*)` references still resolve. If you see `var(--navy)` rendering green, that is why — migrate refs to `--green*` when convenient. Follow-ups: Ben accepted the green provisionally and may want it toned down (change `--green-dark:#0c2c25` and `--green:#15564a` in that one place); the **live `index.html` is still blue** and should get the same treatment before or at cutover (a live change, needs deploy sign-off). Lesson: take brand colours from the actual brand asset (the logo), not a sibling-project template — the ASH/PropOS conventions transfer for engineering discipline, not for visual identity.

**Addendum (4 June 2026):** WCAG AA pass on `feat/pre-cutover-polish`. The teal `--teal:#1a8a7a` was **4.23:1** on white (just under AA 4.5 for small text), so it was darkened to **`#178072` (4.8:1)**; because it is used for both brand accent and text, the one token change fixes section labels, prices, links and the white-on-teal buttons together. Button hovers used to *lighten* to `--teal-light` (white text 2.47:1, a fail) and now **darken** to `--green-mid` (6.4:1). A new `--gold-deep:#8a6d23` (4.9:1) carries gold text on light backgrounds (the home wider-area tag); the dark-hero area badges keep `--gold`/`--teal-light` because light-on-dark already passes.

## L-011 — pgTAP `throws_ok` checks the error MESSAGE, not just the SQLSTATE
The Slice 2 schema test (`supabase/tests/schema_test.sql`) asserts that an overlapping booking is rejected by the `btree_gist` exclusion constraint. Written as `throws_ok(sql, '23P01', 'overlapping non-cancelled job is rejected')` it **failed**: given a third string argument, `throws_ok` treats it as the expected error *message* and compares it to Postgres's actual text. An exclusion-constraint violation emits Postgres's own wording (`conflicting key value violates exclusion constraint "..."`), which never matches the human string, so the assertion failed even though the correct error (SQLSTATE `23P01`) fired. **Fix:** pass `null` as the message (4th arg) so it asserts the SQLSTATE only, and put the human string in the description (5th arg): `throws_ok(sql, '23P01', null, 'overlapping ... is rejected')`. Verified: `supabase test db` passes pgTAP 10/10. Rule: when asserting a specific SQLSTATE in pgTAP, pass `null` for the message unless you genuinely mean to assert Postgres's exact wording.
